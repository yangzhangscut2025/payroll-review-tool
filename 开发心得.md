# 薪酬合规审阅工具 — 开发心得与知识点总结

> 从零到一，一个人+AI 助手，两周完成一个团队协作 Web 应用。

## 一、项目是怎么开始的

需求很简单：薪酬合规团队有一份 Excel 表格，包含各国用工规则，需要对每条规则进行审阅、标注、导出。原来用 Excel 传来传去，版本混乱，不知道谁审了谁没审。

**核心痛点**：
- 多人协作时不知道进度
- 修改历史无法追溯
- 导出格式丢失

**目标**：做一个 Web 工具，上传 Excel → 在线审阅 → 导出结果。

## 二、为什么选 Flask + SQLite

### 什么是 Flask

Flask 是 Python 的 Web 框架。"框架"就是一个脚手架，帮你处理 HTTP 请求、路由、模板渲染这些底层工作，你只需要写业务逻辑。

### 为什么不用 Django

Django 功能更全但更重，自带 ORM、Admin、用户系统。这个项目只需要简单的增删改查，Flask 更轻量，代码量少很多。

### 什么是 SQLite

SQLite 是一个嵌入式数据库，数据存成一个文件（`data.db`），不需要安装数据库服务。MySQL/PostgreSQL 需要单独跑一个服务进程，SQLite 直接读文件就行。

**为什么选 SQLite**：
- 零配置，不需要装数据库
- 8 人以内并发够用
- 备份就是复制一个文件

**代价**：SQLite 不支持高并发写入，8 人同时保存偶尔会 `database is locked`（数据库被锁）。代码里加了 3 次重试解决。

## 三、Excel 解析的坑

### 三种格式的兼容

项目支持三种 Excel 格式（v1/v2/v3），列数不同、列名不同。怎么自动识别？

**思路**：检查表头特征。第一列是「标题Ⅰ」→ v2；第一列是「l1_标题」且没有「内部口径」列 → v3；否则 → v1。

```python
def detect_format(headers):
    if headers[0] == '标题Ⅰ':
        return 'v2'
    if '实务内容（官方）' in headers and '实务内容（内部口径）' not in headers:
        return 'v3'
    return 'v1'
```

### 合并单元格的处理

Excel 里一级模块标题经常跨行合并，openpyxl 读合并单元格时，只有左上角单元格有值，合并区域内的其他单元格为 None。

**解决**：遍历所有合并区域，把左上角的值填充到区域内每个单元格。

```python
merged = {}
for merge_range in ws.merged_cells.ranges:
    top_left_val = ws.cell(row=merge_range.min_row, column=merge_range.min_col).value
    for row in range(merge_range.min_row, merge_range.max_row + 1):
        for col in range(merge_range.min_col, merge_range.max_col + 1):
            merged[(row, col)] = top_left_val
```

### 为什么用 openpyxl 而不是 pandas

pandas 读 Excel 更方便，但它不保留格式（颜色、字体、合并单元格）。这个项目需要导出带格式的 Excel，所以必须用 openpyxl 读写。

## 四、审阅页面的三栏布局

### 布局设计

审阅页面是使用最频繁的页面，布局直接影响工作效率：

```
左侧树(180px) | 主内容(弹性) | Prompt面板(450px)
```

三个区域用 Bootstrap 的 `row` + `col` 实现。左右两栏可折叠。

### 埋过的坑：div 嵌套错误

最初 Prompt 面板一直显示在底部而不是右侧。排查半天发现是 HTML 里多了一个 `</div>`，导致 flex row 提前关闭，Prompt 面板被挤出去。

**教训**：flex 布局的 div 嵌套一定要严格配对，多一个少一个都会导致整个布局错乱。

### 动态宽度计算

面板折叠/展开时，主内容区域的宽度需要动态调整。不能用 CSS 硬编码，因为三个变量叠加：
- 侧栏：0 或 180px
- Prompt 面板：0 或 450px

```javascript
function updateMainColWidth() {
    var sidebarW = getSidebarWidth();  // 0 or 180
    var promptW = promptOpen ? 450 : 0;
    mainCol.style.maxWidth = 'calc(100% - ' + sidebarW + 'px - ' + promptW + 'px)';
}
```

## 五、Quill 富文本编辑器

### 为什么选 Quill

审阅过程中需要标注颜色（红=修改、绿=增加、蓝=无源）和加粗/删除线。普通的 `<textarea>` 只能存纯文本，Quill 可以存带格式的 HTML。

### 颜色白名单

Quill 默认只支持少数颜色。需要手动扩展白名单：

```javascript
var ColorStyle = Quill.import('attributors/style/color');
ColorStyle.whitelist = ['#C00000', '#548235', '#2E75B5', ...];
Quill.register(ColorStyle, true);
```

### 初始化技巧

Quill 编辑器必须在 DOM 中可见才能正确初始化 toolbar。如果容器是 `display:none`，toolbar 绑定会失败。解决方案：先渲染到 `position:absolute; left:-9999px`，初始化完再移回正常位置。

### 导出时保留格式

Quill 生成的 HTML 需要转成 openpyxl 的 RichText 格式。通过正则解析 HTML 标签（`<strong>`, `<span style="color:...">`, `<s>` 等），映射到 openpyxl 的 Font 属性：

```python
# HTML: <span style="color:#C00000;">修改</span>
# openpyxl: InlineFont(color='C00000')
```

## 六、权限系统

### 设计思路

两种角色：**负责人**（创建项目的人）和**校验人**（被分配模块的人）。

- 负责人可以编辑所有模块
- 校验人只能编辑自己被分配的模块

### 实现方式

```python
def check_permission(project, module_l1, username):
    if project.created_by == username:  # 项目创建者
        return True
    assignment = ModuleAssignment.query.filter_by(
        project_id=project.id, module_l1=module_l1).first()
    return assignment and assignment.assignee == username  # 已被分配
```

### 注意事项

权限判断用的是 `project.created_by`（项目创建者），不是用户的角色字段。角色只是用来在 UI 上显示/隐藏按钮，不影响实际权限检查。

## 七、AI Prompt 面板

### 设计理念

审阅时，校验人需要根据参考链接逐条核对内容的准确性。把这个过程交给 AI 可以大幅提效。

**做法**：自动将当前模块的内容和参考链接拼接成一个 Prompt，用户复制到豆包/DeepSeek 对话框里用。

### 为什么推荐复制而不是直接发送

API 调用缺少对话上下文，输出质量不如对话框。而且 API 调用收不到用户的追问，只能一次性返回结果。复制到对话框里，用户可以追问、纠正、要求重新分析。

### Prompt 模板设计

```python
【任务】
请根据我提供的有效链接，逐条核对以下信息的准确性。

【核对规则】
一、逐项判断标准
正确：链接原文能完整支持该信息
错误：信息与链接原文矛盾、数据有误
...

【输出格式】
【信息序号X】判断：[正确/错误/缺漏/未找到/推断成立]
溯源：[链接编号]
说明：

【待核对信息】
<当前模块的内容>

【参考链接】
<链接列表>
```

### v2/v3 格式的链接合并

v2 和 v3 只有 2 个对照对，每个参考依据的链接较少。Prompt 生成时自动合并两个参考面板的全部链接，每个链接标注来源：

```
链接1（官方网站）：https://...
链接2（官方网站）：https://...
链接3（权威网站）：https://...
```

## 八、导出与翻译

### 导出原理

在原 Excel 的每列右侧插入一列「校验列」，填入审阅内容：

```
原始列：实务内容（官方）
校验列：实务内容（官方）_校验列  ← 插入到原始列右边
```

关键：插入列的顺序必须从右到左，否则列号会乱。因为插入一列后，右边所有列的列号都 +1。

### HTML 标签翻译保护

导出时英文 sheet 需要用 AI 翻译审阅内容。但审阅内容里包含 HTML 标签（颜色标注），AI 翻译时语序变化会导致标签错位。

**解决方案**：翻译前把 HTML 标签替换成占位符，翻译后再还原。

```
原文：需要<span style="color:#C00000;">修改</span>的内容
替换：需要{__T0__}修改{__T1__}的内容
翻译：{__T0__}modification{__T1__} needed
还原：<span style="color:#C00000;">modification</span> needed
```

这样 HTML 标签 100% 保留，AI 只翻译纯文本。

### 后台线程导出

导出文件可能很大，翻译耗时几十秒。如果同步处理，前端会一直等待直到超时。

**解决方案**：异步导出 + 进度轮询。

1. 前端发起导出请求 → 后端开一个后台线程处理
2. 前端每秒轮询进度
3. 完成后自动触发下载

```python
thread = threading.Thread(target=run_export, daemon=True)
thread.start()
return jsonify({'ok': True, 'task_id': task_id})
```

**注意**：后台线程没有 Flask 的请求上下文，不能直接操作数据库。需要手动创建 app context：

```python
app = current_app._get_current_object()  # 在主线程拿 app 对象
# 在后台线程中：
with app.app_context():
    db.session.commit()
```

## 九、内网穿透与服务器部署

### 什么是内网穿透

你的电脑在公司内网，没有公网 IP，别人连不上。内网穿透就是用一个公网服务器做中转：

```
你电脑 → 主动连接 → 公网服务器
同事  → 访问公网服务器 → 转发到你电脑
```

### 尝试过的方案

| 方案 | 优缺点 |
|------|--------|
| cpolar | 国内免费，但限速丢包，8 人同时用经常加载不全 |
| ZeroTier | 虚拟局域网，但公司安全软件拦截 |
| bore | 免费不限速，但服务器在新加坡，延迟高 |

### 最终选择：腾讯云服务器

直接把项目部署到云服务器上，省去穿透的麻烦。

**部署步骤**：
1. 买云服务器（腾讯云轻量，2 核 2G，¥65/月）
2. SSH 连接，装 Python 和依赖
3. 上传项目文件
4. 配 systemd 服务（开机自启、崩溃重启）
5. 控制台防火墙放行 5000 端口

**systemd 服务配置**：
```ini
[Unit]
Description=Payroll Review Tool
After=network.target

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/home/ubuntu/review-tool
ExecStart=python3 app.py
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

### Waitress 替代 Flask 开发服务器

Flask 自带的 `app.run()` 是单线程开发服务器，不适合生产环境。Waitress 是生产级 WSGI 服务器，支持多线程：

```python
from waitress import serve
serve(app, host='0.0.0.0', port=5000, threads=16)
```

### 代码同步

本地改代码后，用 `scp` 上传到服务器并重启服务：

```bash
# 打包 → 上传 → 解压 → 重启
scp sync.zip ubuntu@106.53.56.194:/home/ubuntu/review-tool/
ssh ubuntu@106.53.56.194 "cd /home/ubuntu/review-tool && unzip -o sync.zip && sudo systemctl restart review-tool"
```

## 十、学到的经验

### 1. 先跑通核心流程，再打磨细节

一开始只做了上传→审阅→导出，功能少但能用。后续才加 Prompt 面板、翻译、导出预览。

### 2. 前端 JS 别用可选链 `?.`

这个项目同事用各种浏览器，`?.` 语法在旧版 Safari 上不支持。全部用 `var obj = a && a.b && a.b.c` 替代。

### 3. Jinja 模板里 JS 变量用 `|tojson` 转义

```html
<!-- 错误：模块名含特殊字符会导致 JS 语法错误 -->
const CURRENT_L1 = "{{ current_l1 }}";

<!-- 正确：tojson 自动处理转义 -->
const CURRENT_L1 = {{ current_l1|tojson }};
```

### 4. SQLite 备份就是复制文件

不需要 `mysqldump`，不需要 `pg_dump`。直接 `scp data.db` 就是一个完整备份。

### 5. 后台线程不能直接用 Flask 的 `db.session`

Flask 的 `db.session` 和 `current_app` 都依赖请求上下文。后台线程里需要手动创建 app context。

### 6. Python `.format()` 里花括号要转义

```python
# 报错：KeyError: '__T0__'
'翻译时保留 {__T0__} 占位符'.format(items=...)

# 正确：{{ 输出 {，}} 输出 }
'翻译时保留 {{__T0__}} 占位符'.format(items=...)
```

### 7. 导出时插入列要从右到左

```python
# 从右到左排序，插入列号不会乱
sorted_cols = sorted(col_positions.items(), key=lambda x: x[1], reverse=True)
```

### 8. Git worktree 实现多分支并行开发

```bash
git worktree add ../newidea-dev -b dev
```

三个文件夹，三个分支，修改互不影响。

### 9. Quill 编辑器的 `<p>` 标签陷阱

Quill 编辑器保存内容时，会自动给文本包一层 `<p>` 标签。比如用户输入"需要修改的内容"，保存后变成 `<p>需要修改的内容</p>`。

**这导致一个问题**：比较 `changed_content` 和 `original_content` 时，即使审阅人没改任何文字，字符串也不相等（因为 `<p>` 标签）。最初用这个判断哪些字段需要翻译，结果 369 个字段全被判为"有改动"。

**解决**：比较前先剥掉 HTML 标签，只比较纯文本。或者更精确地——只检查有没有审阅标记（`<span style>`、`<s>`、`<strong>`），忽略 `<p>` 和 `<br>`。

### 10. HTML 标签占位符保护

翻译时需要把中文审阅内容译成英文，但审阅内容里有 HTML 颜色标签（`<span style="color:#C00000;">` 等）。AI 翻译时语序变化会导致标签错位。

**解决**：翻译前把 HTML 标签替换成占位符，翻译后再还原。

```
原文：需要<span style="color:#C00000;">修改</span>的内容
替换：需要{__T0__}修改{__T1__}的内容
翻译：{__T0__}modification{__T1__} needed
还原：<span style="color:#C00000;">modification</span> needed
```

注意：Python 的 `.format()` 中 `{` 和 `}` 需转义为 `{{` 和 `}}`，否则会报 KeyError。

### 11. 翻译重试循环

AI 翻译时，偶尔返回的条目数少于输入（比如 20 条进去，19 条出来）。

**解决**：不对整套翻译进行全部丢弃，而是只重试缺失的条目，最多 3 次。重试完仍有缺失的，直接用英文原文作为回退。

### 12. Windows 路径 vs Linux 路径

数据库里的 `file_path` 是 Windows 格式（`C:\Users\...`），部署到 Ubuntu 服务器后找不到文件。

**教训**：部署到不同操作系统时，数据库里的路径要同步更新。本次用 SQL 批量替换修复。

### 13. 两个导出路径需同步维护

项目有两个导出入口：快速导出（`export_quick`）和异步导出（`generate_review_excel`）。它们有各自独立的代码，改一个功能时两边都要改。

**教训**：最好把核心逻辑抽成共享函数，避免维护两份代码。但快速导出为了简洁直接内联了逻辑，导致这次修链接格式时漏了一边。

### 14. 链接格式在 Excel 中的实现

导出时，无效链接需要标红+删除线。openpyxl 支持 `InlineFont` 的 `color` 和 `strike` 属性。把链接包成 HTML 再交给 `_apply_html_to_cell` 即可：

```python
# 无效链接 → 红色+删除线
content = content.replace(url, f'<span style="color:#C00000;"><s>{url}</s></span>')
# 有纠正链接 → 追加
if corrected_url:
    content += f' → {corrected_url}'
```

### 15. 分支保护的实际操作

GitHub 分支保护开启后，直接 push 会被拒绝。日常流程：

```
改代码 → push 到 dev → 开 PR → Review → Approve → Merge
```

**注意**：审批人就是你自己时，去 PR 页面的 Files changed → Review changes → Approve 即可。不需要第二个人。
